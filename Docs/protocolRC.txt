3. ğŸ“± Responsive Design Strategy: Mobile-First Standard

The project's architecture is based on Progressive Enhancement, starting with the smallest screen and scaling up. This is a mandatory professional standard.

Rule

Implementation Standard

Viewport Meta Tag

The public/index.html file MUST include the standard meta tag: <meta name="viewport" content="width=device-width, initial-scale=1.0">.

Base Styling

All initial CSS must target the narrowest viewport (mobile), containing only the essential layout and styles.

Media Queries

MUST exclusively use min-width queries to progressively add complexity for larger screens (e.g., @media (min-width: 768px)). max-width queries are prohibited for main layout adaptation.

Accessibility (A11y)

Development MUST prioritize Semantic HTML (using <header>, <nav>, <main>, etc.) and adhere to ARIA compliance to ensure screen reader compatibility.

4. ğŸ“ Code Standards & Architecture

Area

Rule / Standard

File Naming

PascalCase.jsx for all React components. camelCase.js for all pure JavaScript files (hooks, utilities, services).

Component Architecture

The project uses a Component-Centric Architecture following the principle of Colocation. Every component resides in its own dedicated subdirectory folder.

Component Entry Point (Barrel Pattern)

For all component directories, the folder MUST expose the component via a simple "Barrel Export" file named index.js. The main component file must retain its descriptive name (e.g., ProjectCard.jsx) for developer clarity.

Styling Location

Global (src/index.css): Reserved only for CSS resets, fonts, and global variables. Component-Specific: All styles must use .module.css for strict scoping.

Component Type

All components MUST be written as Functional Components utilizing React Hooks for state and logic.

Tool / Action

7. ğŸ·ï¸ Professional Commit Standard: Conventional Commits

All Git commit messages MUST adhere to the Conventional Commits Specification. This standard ensures a readable, searchable, and automatically parsable project history, which is mandatory for professional software engineering.

ğŸ“ Mandatory Commit Message Structure

Commit messages must use the following structure: type(scope): description

Key Types and Explanation

Type

Purpose

Short Explanation

feat

Features/Enhancements

Used when adding a new capability, component, or enhancement. (e.g., feat(ui): Add mobile navigation menu)

fix

Bug Fixes

Used for all code changes that correct a defect or unexpected behavior.

refactor

Code Restructuring

Used for code changes that neither fix a bug nor add a feature, but improve code structure, quality, or readability.

chore

Maintenance

Used for routine tasks, dependency updates, and minor build/tooling changes. (e.g., chore(deps): Update Prettier version)

style

Formatting

Used for changes that only affect code style (whitespace, semicolons, missing semicolons, etc.) and not logic.

docs

Documentation

Used for changes solely related to documentation, like READMEs or protocol updates.

test

Testing

Used when adding or correcting unit tests or integration tests.
