3. üì± Responsive Design Strategy: Mobile-First Standard

The project's architecture is based on Progressive Enhancement, starting with the smallest screen and scaling up. This is a mandatory professional standard.

Rule

Implementation Standard

Viewport Meta Tag

The public/index.html file MUST include the standard meta tag: <meta name="viewport" content="width=device-width, initial-scale=1.0">.

Base Styling

All initial CSS must target the narrowest viewport (mobile), containing only the essential layout and styles.

Media Queries

MUST exclusively use min-width queries to progressively add complexity for larger screens (e.g., @media (min-width: 768px)). max-width queries are prohibited for main layout adaptation.

Accessibility (A11y)

Development MUST prioritize Semantic HTML (using <header>, <nav>, <main>, etc.) and adhere to ARIA compliance to ensure screen reader compatibility.

4. üìù Code Standards & Architecture

Area

Rule / Standard

File Naming

PascalCase.jsx for all React components. camelCase.js for all pure JavaScript files (hooks, utilities, services).

Component Architecture

The project uses a Component-Centric Architecture following the principle of Colocation. Every component resides in its own dedicated subdirectory folder.

Component Entry Point (Barrel Pattern)

For all component directories, the folder MUST expose the component via a simple "Barrel Export" file named index.js. The main component file must retain its descriptive name (e.g., ProjectCard.jsx) for developer clarity.

Styling Location

Global (src/index.css): Reserved only for CSS resets, fonts, and global variables. Component-Specific: All styles must use .module.css for strict scoping.

Component Type

All components MUST be written as Functional Components utilizing React Hooks for state and logic.

Standard

Tool / Action

7. üè∑Ô∏è Professional Commit Standard: Conventional Commits

All Git commit messages MUST adhere to the Conventional Commits Specification.
This standard ensures a readable, searchable, and automatically parsable project history,
which is mandatory for professional software engineering.

üìù Mandatory Commit Message Structure

Commit messages must use the following structure: type(scope): description

Key Types and Explanation

Type

Purpose

Short Explanation

feat

Features/Enhancements

Used when adding a new capability, component, or enhancement. (e.g., feat(ui): Add mobile navigation menu)

fix

Bug Fixes

Used for all code changes that correct a defect or unexpected behavior.

refactor

Code Restructuring

Used for code changes that neither fix a bug nor add a feature, 
but improve code structure, quality, or readability.

chore

Maintenance

Used for routine tasks, dependency updates, and minor build/tooling changes. (e.g., chore(deps): Update Prettier version)

style

Formatting

Used for changes that only affect code style (whitespace, semicolons, missing semicolons, etc.) and not logic.

docs

Documentation

Used for changes solely related to documentation, like READMEs or protocol updates.

test

Testing

Used when adding or correcting unit tests or integration tests.

üìÑ Protocol Addition: Professional Note-Taking
Mandate: For all new code and major revisions, the codebase MUST incorporate professional, context-providing comments to support learning and maintainability.

Requirements:
JSDoc: Every function/component (const Component = () => {}) MUST be immediately preceded by a JSDoc block to document its purpose and expected outputs.

Intent Comments: All architectural decisions (e.g., Barrel Export syntax, complex regex, non-obvious API calls) MUST be annotated with a // WHY: comment explaining the rationale.

üìã Git Professional Protocol & Standard Setup
A. Initial Git Setup (Standardized Configuration)
These steps ensure consistent identity and line endings to prevent common conflict types.

Set Global User Name:

Markdown

git config --global user.name "Your Name"
Set Global User Email:

Markdown

git config --global user.email "your.email@example.com"
Set Line Endings (CRLF vs LF) - Crucial for Cross-OS Work:

macOS/Linux:

Markdown

git config --global core.autocrlf input
Windows:

Markdown

git config --global core.autocrlf true
Set Default Merge Tool (Optional, for advanced users):

Markdown

git config --global merge.tool meld
B. Conflict Resolution Action Plan
Protocol for resolving conflicts during rebase or merge operations.

Secure State & Document (Professional Notes):

git status (Confirm operation & list conflicted files)

Professional Note: Document: Conflict Source:, Conflicted Files:, Goal:

Manual Resolution & Staging:

Open all conflicted files.

Resolve manually by removing all Git conflict markers (<<<<<<<, =======, >>>>>>>).

Stage each resolved file:

Markdown

git add <path/to/resolved/file>
Resumption & Final Push:

Resume operation:

Markdown

git rebase --continue
# OR
git merge --continue
If history was rewritten (e.g., post-rebase), use a safe force push:

Markdown

git push --force-with-lease
üìÖ Daily Startup Procedure
This mandatory routine ensures the local development environment is always synchronized with the remote repository and ready for professional development.

Navigate to Project Directory:

Markdown

# Assume you are in the correct directory (e.g., react-projects/portfolio2023-main)
# cd ~/react-projects/portfolio2023-main
Synchronize Code Base (Mandatory):

Always pull the latest changes from the remote to ensure your work is built on the most current code.

Markdown

git pull origin main
Rationale: Prevents "Pull Before Push" conflicts, minimizes future rebase complexity, and ensures local code is synchronized.

Check for Dependency Updates:

If git pull reports changes to package.json or package-lock.json, or if you are starting a fresh day, run an install to synchronize dependencies.

Markdown

npm install
# OR (Professional standard for CI/clean installs)
# npm ci 
Launch Development Environment:

Start the local development server to begin work.

Markdown

npm run dev
